## Updating Shared Reference

**`validated-facts-org.md`**: Add a fact here only after it is confirmed in 2+ member projects. Include provenance: which projects, which dates, what evidence. Do not add single-project observations — those stay in the project's `.codex/reference/validated-facts.md`.

**`llms.txt`**: Update when:
- A new project joins the org (add entry under `## Projects`)
- A project is archived or removed (remove or mark entry)
- Repository paths change
- A new cross-repo initiative starts or completes (update `## Active Initiatives`)

**`org-coordination-guide.md`** (this file): Update when the coordination conventions themselves change (new directory, new maintenance rule, new workflow).

## Org vs. Project Scope Decision Rule

When deciding whether content belongs in this org repo or in a project's `.codex/`:

**Org scope** — content belongs here when:
- The fact or pattern applies to 2+ member projects (graduation rule from D80 § 4)
- It is a cross-project coordination concern (shared service registry, org-level initiative)
- It is an org-level command used across multiple projects (`/health-check`, `/intake`)
- It is a cross-repo architecture decision (org-level ADR)

**Project scope** — content stays in `[project]/.codex/` when:
- It is project session state or project backlog (always project-scoped)
- It is a project-specific deployment configuration or environment variable
- It is a fact validated in only one project (promote to org after second confirmation)
- It is a project-specific artifact, test fixture, or retrospective

When in doubt: start per-project. Graduate to org scope after the second project confirms the same pattern.

## Adding a New Project to the Org

1. **Set up the project's `.codex/` scaffold**: Follow the per-project scaffold steps (steps 01–10) in that project's repo.
2. **Add entry to `llms.txt`**: Under `## Projects`, add a line: `[REPO_NAME]: [path] — [one-line description]`.
3. **Link from the project**: In the project's manager protocol file (`CONDUCTOR.md`), add an `## Org Context` section: `This project is part of [ORG_NAME]. Org-level coordination: [ORG_REPO_PATH]`.
4. **Audit for graduation candidates**: Review the new project's `validated-facts.md` — if any facts overlap with existing member project facts, consider graduating them to `validated-facts-org.md`.
5. **Update active initiatives**: If the new project is involved in any existing cross-repo initiative, update `llms.txt` `## Active Initiatives`.

## Health Check Cadence

Run `/health-check` from this repo:
- **Monthly**: routine verification that all project paths resolve and `validated-facts-org.md` is fresh
- **After any project is added or removed**: confirm `llms.txt` is accurate
- **After any org restructuring**: confirm paths, entries, and gitignore rules are correct

**Stale state signals**:
- `validated-facts-org.md` unmodified for >30 days without a "stable" marker — review for freshness
- `llms.txt` lists a path that no longer resolves — update or remove the entry
- `state/session.md` is tracked by git (not gitignored) — immediate fix required (see `.gitignore`)
- Org `BACKLOG.md` has items marked "In Progress" with no owner for >14 days — re-triage

**When health check finds issues**: Address individually. The health check is read-only — it observes, does not fix. Fix issues manually after reading the report.
