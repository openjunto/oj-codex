# /intake (Org Scope)

Triage an inbound request and route it to the correct backlog: org-level (this repo) or a specific member project's backlog.

**Constraint**: This command does NOT write to per-project `.codex/` files. The org repo has no authority over project files. Routing is advisory — the manager drafts the backlog item; the user carries it to the right project.

## Steps

### 1. Capture the Request

If the user has not provided an inbound request in their message, ask:

> "What is the inbound request or task you want to triage? Please describe what needs to be done and any relevant context."

Once the request is captured, read the following for context:
- `BACKLOG.md` — current org-level backlog (check for duplicates or related items)
- `llms.txt` — member project list (needed for routing in step 2)

### 2. Scope Triage

Determine whether this is org-level work or single-project work:

**Org-level work** — route to org backlog when:
- The request affects 2 or more member projects
- It is a coordination concern (cross-project dependency, shared service change, org process)
- It involves shared infrastructure or shared validated facts
- It requires an org-level architecture decision

**Single-project work** — route to a project backlog when:
- The request is scoped to one specific project
- It is implementation work within a single repo
- It is project-specific state, config, or artifact

If unclear, default to the narrowest scope (project) and note the ambiguity in the routing decision.

If single-project: identify which project from `llms.txt`. If the relevant project is not listed, note this and suggest adding it to `llms.txt`.

### 3. Standard OpenJunto Triage

Apply the two-dimensional triage to the scoped request:

**Execution model** — how should this work be done?
- Agent-executable: well-defined, bounded, clear completion criteria
- Human-executable: requires judgment, access, or stakeholder action
- Collaborative: agent + human working together

**Stakeholder identification** — who needs to be involved or informed?
- Identify the requestor, approver, and any affected teams
- Note if cross-project stakeholders need to be looped in

### 4. Route and Draft

**If org scope**: Draft the BACKLOG.md item for the org backlog:

```markdown
### BACK-NNN: [Title]
**Priority**: [High | Medium | Low]
**Status**: Open
**Created**: [TODAY'S DATE]
**Scope**: org-level — affects [list affected projects]
**Execution**: [Agent-executable | Human-executable | Collaborative]
**Stakeholders**: [requestor, approver, affected teams]
**Description**: [What and why, one paragraph]
```

**If project scope**: Draft the BACKLOG.md item for the target project's backlog (do NOT write it there — provide the text for the user to add):

```markdown
### BACK-NNN: [Title]
**Priority**: [High | Medium | Low]
**Status**: Open
**Created**: [TODAY'S DATE]
**Execution**: [Agent-executable | Human-executable | Collaborative]
**Stakeholders**: [requestor, approver]
**Description**: [What and why, one paragraph]
```

Note the target: "Add this item to `[REPO_NAME]/.codex/BACKLOG.md`" (or the project's equivalent backlog path).

### 5. Confirm Before Writing

Present the routing decision and draft backlog item to the user:

```
Routing decision: [org | project: REPO_NAME]
Reason: [one sentence explaining why]

Draft backlog item:
[the drafted item text]

Shall I add this to [org BACKLOG.md | the [REPO_NAME] backlog (advisory — you will add it)]?
```

Wait for user confirmation before writing anything.

**If confirmed for org backlog**: Append the item to `BACKLOG.md` under `## Active`. Do not modify any other file.

**If confirmed for project backlog**: Do NOT write to the project repo. Provide the final item text for the user to copy. Confirm: "Please add this to `[path]/BACKLOG.md` in the [REPO_NAME] repo."

## Constraints

- NEVER write to per-project `.codex/` files. The org intake command is advisory for project-scoped items.
- NEVER write to `BACKLOG.md` before user confirmation in step 5.
- Do not load per-project `.codex/` directories beyond what is listed in `llms.txt`.
- If routing is ambiguous, present both options and ask the user to decide before drafting.
